⚑ Safety rules — non-negotiable

Data isolation

  • Production never touched. All data → the cloud test project (or emulator), never fir-sample-aae4a.
  • Test users live only there — never in production.
  • Watson / SalesCRM are separate prod apps via getApp(); the test env wires no cross-project writer.

ATC & config segregation

  • ATC off-limits. No suite reads/writes/seeds ATC collections; products seeded atcmodel:null.
  • The one ATC-named test is a pure render smoke over a synthetic array — no Firestore, no ATC import.
  • Emulator config lives in firebase.emulator.json / firebase.test.json, not firebase.json (a bare firebase deploy would push to prod).

$ How to run

Prerequisites (one-time):

# repo deps (peer deps require the flag) npm install --legacy-peer-deps # the e2e runner lives here (Playwright 1.60 — NOT the repo-root 1.59) cd e2e && npm install

The app must be served on :4200 (test-project dev build). The configs auto-start it; to (re)build yourself:

node_modules/.bin/ng build --configuration development # development env → test project

Run it — bundled npm scripts (from e2e/):

# queue suite npm run test:cloud # queue vs the cloud test project (~188 cases) npm run test:emu # queue vs the local Firebase emulator (hermetic, CI-gateable) # any single component group npm run test:appointments # · events · modes · content · workshops · comms npm run test:profiles # · support · evomap · authroles · journey · business npm run test:groups # ALL 12 component groups in sequence (~40 min) npm run test:invariants # hermetic oracle self-test (no app, no seed) npm run test:mobile # REAL Flutter participant walk on the iOS simulator # seed / evidence npm run seed:cloud # (re)seed the cloud project manually npm run report:cloud # isolated run + per-stage screenshots + HTML report

Conferencing / video tiles are root scripts (from the repo root):

npm run e2e:T1 # grid … npm run e2e:T10 # rejoin (T1-grid, T2-join-leave, T3-camera, T4-mic, T5-network, T6-bg, T7-screenshare, T8-blur, T9-sustained, T10-rejoin)

🔍 Playwright CLI — view reports, debug, capture screenshots

Every suite auto-records on failure: screenshots (screenshot: 'only-on-failure'), a trace on the first retry (trace: 'on-first-retry'), and a self-contained HTML report at playwright-report-<group>/. Anything after -- is forwarded to Playwright, so it composes with any npm run test:<group> (commands from e2e/):

# ── view results (screenshots embedded on failed steps) ── npx playwright show-report playwright-report-journey # opens the web report on :9323 npx playwright show-trace test-results/<dir>/trace.zip # DOM/network/console time-travel # ── watch it run / step through ── npm run test:journey -- --headed # real browser, visible npm run test:journey -- --ui # time-travel UI (watch + re-run + pick locators) npm run test:journey -- --debug # Playwright Inspector, pause & step # ── force-capture artifacts even on green (or: npm run report:cloud:full) ── npm run test:journey -- --trace=on --screenshot=on --video=on # ── target / filter / flake-hunt ── npm run test:journey -- journey/catalog.spec.ts:132 # one test by file:line npm run test:journey -- -g "JP-04" # by title substring npm run test:journey -- --last-failed # re-run only last run's failures npm run test:journey -- --repeat-each=5 # run a case 5x to surface flake # ── one-time browser install ── npx playwright install chromium

Reports live per group: playwright-report-{appointments,events,modes,…,business}/, queue → playwright-report/ (+ results.xml JUnit), mobile → playwright-report-mobile/. The report:cloud / report:cloud:full npm scripts wrap an isolated run with EVIDENCE=1 (per-stage screenshots) / TRACE=full.

! Caveats — read before trusting a run

  1. Shared cloud project → occasional timing flake. Every suite hits one reused project that accumulates state. A capacity/auth-timing case can flake ~1/run on a busy day. Re-run before treating a single failure as real. Each suite reseeds its own world on start.
  2. :4200 can serve a STALE bundle from a sibling worktree. Testing new/uncommitted source? The default server may be another worktree's old build. Build this repo, serve on a fresh port, pass BASE_URL. A serve -s SPA fallback returns 200 for any path — confirm content_type: application/javascript, not the HTML fallback.
  3. Serial by design. Each config runs one worker (shared seed state) — don't add --workers.

⚙ Shared harness — to add a new group

  • e2e/lib/seed-common.js — bootstrapGroup / seedDashboardRoutes / teardownGroup. Every driven route needs a dashboard route-grant doc or the authGuard redirects to root.
  • e2e/_shared/prod-firewall.ts — installProdFirewall(page) blocks hardcoded prod CF URLs.
  • Reuse queue support/{console-guard,firestore-admin} + stubs/* + actors.loginAs.
  • assertNoFatal(guard, ctx, extraIgnorable[]) — 3rd arg tolerates a tightly-anchored benign console class per screen.
  • Composite indexes → firestore.indexes.json (firebase deploy --only firestore:indexes --config firebase.test.json).
